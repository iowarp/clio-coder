The user has tested polishes #1 and #2 interactively and reports the tool

### What the user sees today

### What pi-coding-agent does (reference: `~/tools/pi-mono/packages/coding-agent/src/modes/interactive/components/tool-execution.ts`)

`ToolExecutionComponent extends Container` â€” it's a **stateful component
hierarchy**, not a string-list renderer. It composes:

- A `Box` with padding and a tinted background for the body region.
- A `Text` header with theme-driven coloring.
- Per-tool `callRendererComponent` and `resultRendererComponent` chosen
  from a **renderer registry** keyed by tool name. Each renderer can be a
  full pi-tui `Component` subtree.
- An `Image` array for image results (Kitty / iTerm2 inline image
  protocols).
- An `expanded` toggle (`Ctrl+O` in pi-coding-agent) that switches between
  collapsed summary and full body.
- `partial` state during streaming so the tool block updates in place
  rather than producing a fresh string list every paint.

The pi-tui primitives Clio's engine boundary already re-exports
(`src/engine/tui.ts`):

- `Box` â€” padded container with optional bg fn.
- `Container` â€” generic stack of children.
- `Text` â€” single styled string.
- `TruncatedText` â€” single line with width-aware truncation.
- `Markdown` â€” full markdown component (could render `read README.md`
  as actual styled markdown!).
- `Image` â€” inline image (Kitty/iTerm2/sixel).
- `Spacer` â€” vertical gap.
- `Loader` / `CancellableLoader` â€” animated progress.
- `Editor`, `Input`, `SelectList`, `SettingsList` â€” interactive forms.
- `wrapTextWithAnsi`, `truncateToWidth`, `visibleWidth` â€” width helpers.

Clio currently uses **none** of these for the tool block. The renderer
returns `string[]`; the chat-panel splices those strings into its own
`appendReplayBlock` mechanism. That's why the result reads as flat indented
text instead of as a "tool capsule" with its own visual identity.

### The redesign brief

Use the `frontend-design` skill (load it via Skill tool) to design a
**tool capsule** for Clio's TUI that:

1. Reads as a single visual object distinct from chat prose. The user
   should be able to scan a transcript and instantly tell "this is a tool
   call" vs "this is the agent talking" without parsing words.
2. Carries an unmistakable success/failure signal (today's `âś“`/`âś—` glyph
   is OK; consider a colored left border block, e.g. `â–Ž` (U+258E) painted
   green/red, that runs the full height of the body â€” much more legible
   at a glance than a single character at the end of one line).
3. Has tool-specific result rendering. Today every tool's result is
   indented text. Better:
   - `read` â†’ if the file is markdown, render through `Markdown`. If it's
     code, render through a syntax-highlighter (we already depend on
     `cli-highlight` per the parity spine Â§5.1).
   - `edit` â†’ already uses the diff renderer, keep but visually integrate
     into the capsule.
   - `bash` â†’ command line on its own row with a `$` prefix; output below
     using a monospace block style.
   - `web_fetch` â†’ URL + status code on the header; body rendered like
     `read` (markdown if html/markdown, raw otherwise).
   - `grep` / `glob` â†’ file:line list, paths colorized.
   - Unknown tools â†’ fall back to today's behavior.
4. Supports a collapsed/expanded state. Default: collapsed summary
   (header + first 3 lines + line-count footer). User toggles to full
   with a key (e.g. `Ctrl+O` per pi-coding-agent's convention; pick a
   key that doesn't collide with existing Clio bindings â€” check
   `src/domains/config/keybindings.ts`). When expanded, the 12-line cap
   relaxes to a much higher cap or to the full output.
5. Streams in place during `tool_execution_update`. Today our renderer
   ignores partial results (the chat-panel's update handler is a
   `markDirty()` no-op with an explanatory comment). The redesign should
   surface partial output in the body so the user sees a long bash
   command's output stream live.

### Architectural shift required

Doing this with pure string functions (today's
`renderToolExecution(...)`) hits a ceiling. To match pi-coding-agent's
quality you almost certainly need to:

- Introduce a `ToolCapsule` component (a `Container` subclass) owned by
  the chat-panel.
- Define a **per-tool renderer registry** in `src/interactive/renderers/
  tool-renderers/` with one file per tool (`read.ts`, `bash.ts`,
  `edit.ts`, `web_fetch.ts`, etc.) each exporting a factory that returns
  a pi-tui `Component` for the result region.
- Move the chat-panel's `tool_execution_*` handlers to construct, update,
  and finalize a `ToolCapsule` instead of appending string blocks.
- Keep `tool-execution.ts` as the fallback string-list renderer for the
  replay path on session resume (until that path is also upgraded).

This is a multi-slice effort. Plan it as its own `docs/superpowers/plans/`
file before touching code. Brainstorm with `superpowers:brainstorming`
first if scope is unclear.
