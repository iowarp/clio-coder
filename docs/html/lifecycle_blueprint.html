Installation Methods

Method A: Global NPM (Published)

Standard global installation from the NPM registry. Places files into global node modules and exposes `clio` link in paths. 

npm install -g @iowarp/clio-coder
Method B: Local Checkout (Development)

Checkout the repository directly, build the TypeScript binary bundle locally, and register a local executable link. 

git clone https://github.com/iowarp/clio-coder.git
cd clio-coder
npm ci
npm run build
npm link
Developer Integration: Running npm run hooks:install copies the pre-commit git hook script into .git/hooks/pre-commit. It triggers npm run ci (typechecking, Biome format, and smoke tests) before commits.

First-Run Bootstrap (`initializeClioHome`)

When the application starts, it performs an idempotent bootstrap operation inside src/core/init.ts to prepare the user filesystem:

  • 1

    Directory Creation

    Creates config, data, and cache roots recursively if missing, followed by the 9 default data subfolders.

  • 2

    Default Config Generation

    Writes default settings.yaml containing base runtimes and client structures (mode: 0o644).

  • 3

    Credentials Template

    Writes credentials.yaml with comments, locking its permissions down to owner-only read-write (mode: 0o600).

  • 4

    Install Registry

    Writes metadata (Clio version, node version, platform, install timestamp) inside data/install.json.

Diagnostic Checks (`clio doctor`)

Clio includes a doctor utility (src/domains/lifecycle/doctor.ts) to verify the integrity of configurations, file formats, and permissions. Here is the list of verification runs:

Check Target Details & Validations Recovery Action (`--fix`)
Permissions Verifies credentials.yaml permissions are locked at 0o600 (read-write by owner only). Warns if world-readable. Resets chmod permissions to 0o600.
Config Format Parses settings.yaml as valid YAML to ensure no user formatting syntax errors are present. If missing entirely, recreates it using standard template.
Node & Packages Ensures Node.js ≥ 22.19 and verifies active bindings for core agent dependencies. Prints package manager commands to resolve.
Drift Detection Compares version in install.json with active execution binary. Detects version upgrades. Rewrites install.json with upgraded version info.
Legacy Runtimes Scans settings for deprecated adapter strings (e.g. llamacpp-completion). Migrates settings inline to current unified adapters (e.g. llamacpp).

Upgrade Mechanism (`clio upgrade`)

Upgrading Clio Coder updates both the binary (via npm global installation hooks) and its active database directories. The command performs a coordinated upgrade in two phases:

Phase 1: Binary Upgrade

The running binary spawns npm to install the latest bundle from the registry or specified channels (latest, beta, dev). 

# Clio spawns:
npm install -g @iowarp/clio-coder@latest
Phase 2: Post-Install Actions

Clio spawns its newly installed version to migrate the state folder and correct environment differences. 

# Clio spawns:
clio upgrade --post-install
clio doctor --fix

Durable State Migrations

State-shape transformations (configured inside src/domains/lifecycle/migrations/) are versioned, incremental modules compiled into the distribution bundle.

Manifest Registry: state/migrations.json

Clio writes all successfully applied migration unique string keys into a manifest file. During future upgrades, Clio reads the manifest, compares it with its internal catalog, and runs pending scripts only. 

{
  "applied": [
    "2026-04-17-initial"
  ]
}

Migration Rule Execution Rules

  • Stable Ordering: Migrations are keyed with stable IDs like YYYY-MM-DD-<slug> and executed chronologically.
  • Atomic Execution: If a migration script fails, the manifest is not updated, preventing partial migration corruption.
  • No File Sweeps: The registry is statically built, avoiding runtime filesystem sweeps for migration files.

Clean Uninstallation (`clio uninstall`)

The clio uninstall command handles removing state directories, cache files, and configs from the system:

  • 1

    State Deletion

    Deletes the resolved data directory, configuration files, and temporary cache folders according to user flags.

  • 2

    Binary Clean Guidance

    Instructs the user on how to delete the global binary based on their original package management choice.

# Remove state (requires --force)
clio uninstall --force

# Preview removal actions without writing changes
clio uninstall --dry-run

# Preserve configs (settings/keys) while wiping data
clio uninstall --keep-config --force

Selective Reset Wipes (`clio reset`)

If Clio gets corrupted or you want to start fresh, clio reset wipes selective scopes. All options support --dry-run and require --force.

Command Flag Wipe Scope Resulting State
--state (default) Removes data/ and cache/. Clears sessions, audits, memory. Retains target configs.
--auth Removes credentials.yaml. Wipes all saved target API keys and OAuth tokens.
--config Removes settings.yaml. Resets settings to defaults. Keeps session databases.
--all Removes all roots (config, data, cache). Complete wipe. Re-runs initialization automatically.

System Clean Up & Leftover Checklist

To ensure 100% cleanliness, check these locations for residues when performing manual removals:

1. Configuration / Credentials / Cache

Paths containing global setup variables and local files:

  • ~/.config/clio/ (settings, keys)
  • ~/.local/share/clio/ (session histories, audits)
  • ~/.cache/clio/ (temp cache files)

2. Repository Checkouts (Git Hooks)

Any checkout repository that had git integrations enabled:

  • <your-repo>/.git/hooks/pre-commit (removable via rm)

3. Global Binary Links

Global symbolic links registered in package directories:

  • NPM prefix: /usr/local/bin/clio or ~/.npm-global/bin/clio
  • Linked modules: node_modules/@iowarp/clio-coder link
