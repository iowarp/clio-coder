Clio Coder Architecture Specifications

Clio Coder is an experimental, terminal-first coding harness for the CLIO ecosystem. Its architecture favors small, auditable subsystems over a single monolithic agent loop: CLI entry points, the interactive TUI, provider/runtime code, worker subprocesses, tools, and feature domains are kept separate so local-model support and scientific-software workflows can evolve without collapsing safety boundaries.

Source Code Layout

Folder Responsibility
src/cli/ clio subcommands, argument parsing, headless run modes
src/core/ config/state paths, event bus, defaults, shared primitives
src/domains/ feature domains loaded through manifests/contracts
src/engine/ pi-ai/provider boundary and runtime adapters
src/entry/ orchestrator bootstrap wiring
src/interactive/ TUI panels, overlays, key routing, dashboard, slash commands
src/tools/ built-in tool specs and registry admission boundary
src/worker/ subprocess worker entry/runtime rehydration

Boundary Invariants

Boundary rules are test-enforced by npm run check:boundaries. Three rules are strictly enforced:

  • Rule 1: pi-ai value imports stay in src/engine/**
    Only files under src/engine/** may value-import `@earendil-works/pi-*` packages. Outside `src/engine`, only type-only imports are allowed.
  • Rule 2: workers do not value-import domains except runtime rehydration
    src/worker/** may not value-import `src/domains/**` except the worker-safe provider runtime rehydration modules (plugins.ts, registry.ts, runtimes/builtins.ts).
  • Rule 3: domains do not import each other's `extension.ts`
    A file under src/domains/<x>/** must not import `src/domains/<y>/extension.ts` for `y != x`.

Command Verification

npm run check:boundaries     # check boundaries rule suite
npm run typecheck            # type check TS
npm run test                 # run tests suite
npm run build                # compile build package
