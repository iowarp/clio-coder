Resource Precedence & Scoping Hierarchy

Clio Coder resolves resource conflicts deterministically using ranked precedence tiers. Native project configurations outrank user configurations, which outrank compatibility layers and package-contributed roots.

Prompts Precedence Tiering

Rank Scope Origin Source Default Directory Path
0 package Enabled extension resource directories <configDir>/extensions/<id>/prompts
1 user Clio User Prompts Folder ~/.config/clio/prompts
2 project Clio Project Local Prompts Folder .clio/prompts
3 cli Call-site Injected Templates Direct inline strings or temporary buffers

Skills Precedence Tiering

Skills add Agent Skills compatibility roots, allowing shared access across various agent frameworks. Native Clio folders outrank compatibility ones at the same scope.

Precedence Scope Agent Category Standard Directory Path Model Visibility
10 package Extension <configDir>/extensions/<id>/skills Visible (if enabled)
20 user Compatibility (agents/claude/codex/etc.) ~/.agents/skills, ~/.claude/skills, etc. Visible
30 user Clio Native User ~/.config/clio/skills Visible
40 project Compatibility (agents/claude/codex/etc.) .agents/skills, .codex/skills, etc. Untrusted by default (requires opt-in)
50 project Clio Native Project .clio/skills Visible
60 cli Explicit Paths CLI parameter --skill <path> Visible (Forced injection)

Extension Package Layout

Extension roots package reusable skills and prompts. The layout follows standard modular separation:

clio-kit/                  # Ext package root
  clio-extension.yaml      # Package manifest
  prompts/                 # Contributed prompt templates
    hdf5-fix.md
  skills/                  # Contributed custom skills
    scientific-audit/
      SKILL.md             # Standard skill md
      scripts/             # Custom auxiliary scripts
      references/          # References files

Diagnostics & Safety

Untrusted project roots: Shared local project compatibility roots (like .agents/skills) are deactivated by default to prevent arbitrary code/instruction injections. Opt-in is configured via skills.trustProjectCompatRoots: true in settings.yaml or setting the environment variable CLIO_TRUST_PROJECT_SKILLS=1.

Graceful Degradations: Only a missing description inside a skill frontmatter is a hard rejection. All other validation anomalies (consecutive hyphens in names, name-directory mismatches, frontmatter format failures) degrade to warnings, enabling the skill or template to still load with diagnostics.
